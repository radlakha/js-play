<div class="q">When should you prefer <code>interface</code> over <code>type</code> for defining a <code>Task</code> entity?</div><div class="options"><div class="option"><span class="opt-label">A)</span> When the shape will never need optional properties</div><div class="option"><span class="opt-label">B)</span> When you need to define a union like <code>string | number</code></div><div class="option"><span class="opt-label">C)</span> When defining an object shape that represents a real entity, may grow over time, and could be extended &mdash; <code>interface</code> is idiomatic here</div><div class="option"><span class="opt-label">D)</span> <code>interface</code> and <code>type</code> are completely interchangeable &mdash; purely a style preference</div></div>	<div class="answer">C &mdash; <code>interface</code> for entities; <code>type</code> for unions and complex compositions</div>	ch06 types interfaces
<div class="q">What does this destructuring syntax do?</div><pre>const { title, completed } = task;</pre><div class="options"><div class="option"><span class="opt-label">A)</span> Creates a shallow copy of the entire <code>task</code> object</div><div class="option"><span class="opt-label">B)</span> Extracts <code>title</code> and <code>completed</code> into their own variables, without modifying <code>task</code></div><div class="option"><span class="opt-label">C)</span> Removes <code>title</code> and <code>completed</code> from the <code>task</code> object</div><div class="option"><span class="opt-label">D)</span> Only works if <code>task</code> is typed as <code>any</code></div></div>	<div class="answer">B &mdash; extracts named properties into local variables; task is unchanged</div>	ch06 interfaces destructuring
<div class="q">What does the spread produce in this object update pattern?</div><pre>const updatedTask = {<br>    ...task,<br>    completed: true<br>};</pre><div class="options"><div class="option"><span class="opt-label">A)</span> <code>updatedTask</code> is the same object as <code>task</code> &mdash; spread is a reference, not a copy</div><div class="option"><span class="opt-label">B)</span> A new object with all of <code>task</code>'s properties, but <code>completed</code> overridden to <code>true</code> &mdash; <code>task</code> is unchanged</div><div class="option"><span class="opt-label">C)</span> TypeScript will error &mdash; you cannot override a property after a spread</div><div class="option"><span class="opt-label">D)</span> The spread copies nested objects by value, creating a deep clone</div></div>	<div class="answer">B &mdash; new object, <code>completed</code> overridden, original <code>task</code> untouched</div>	ch06 interfaces immutability
<div class="q">What does the <code>?.</code> operator do here?</div><pre>console.log(task.dueDate?.toISOString());</pre><div class="options"><div class="option"><span class="opt-label">A)</span> Marks <code>dueDate</code> as optional in the interface</div><div class="option"><span class="opt-label">B)</span> A ternary &mdash; returns <code>toISOString()</code> if truthy, <code>undefined</code> if falsy</div><div class="option"><span class="opt-label">C)</span> Optional chaining &mdash; if <code>dueDate</code> is <code>undefined</code> or <code>null</code>, the expression short-circuits and returns <code>undefined</code> instead of throwing</div><div class="option"><span class="opt-label">D)</span> Converts <code>dueDate</code> to a string automatically</div></div>	<div class="answer">C &mdash; optional chaining; prevents "cannot read property of undefined" errors</div>	ch06 interfaces
<div class="q">What does the rest syntax collect into <code>rest</code>?</div><pre>const { id, ...rest } = task;</pre><div class="options"><div class="option"><span class="opt-label">A)</span> An array of the remaining values</div><div class="option"><span class="opt-label">B)</span> An object containing all properties of <code>task</code> <em>except</em> <code>id</code></div><div class="option"><span class="opt-label">C)</span> A string representation of the remaining properties</div><div class="option"><span class="opt-label">D)</span> Invalid syntax &mdash; rest syntax only works with arrays, not objects</div></div>	<div class="answer">B &mdash; all properties except <code>id</code>, as a new object</div>	ch06 interfaces destructuring
<div class="q">When should you use <code>as</code> (type assertion)?</div><pre>const task = rawData as Task;</pre><div class="options"><div class="option"><span class="opt-label">A)</span> Whenever you want to change the runtime type of a value</div><div class="option"><span class="opt-label">B)</span> As a first resort &mdash; it is the safest way to type-check data</div><div class="option"><span class="opt-label">C)</span> Sparingly &mdash; only when TypeScript cannot infer the type correctly, since it bypasses type checking and can hide real bugs</div><div class="option"><span class="opt-label">D)</span> Only when working with <code>string</code> types</div></div>	<div class="answer">C &mdash; last resort only; it silences the type checker without adding safety</div>	ch06 types

<span class="concept-label">Concept</span><div class="q">The spread operator creates a <em>shallow copy</em>. What does that mean, and when does it matter?</div>	<div class="concept-answer"><b>Shallow Copy vs Deep Copy</b><br><br>A <b>shallow copy</b> copies top-level properties by value &mdash; but nested objects and arrays are still <b>shared by reference</b>.<br><br><pre>const updated = { ...task, completed: true };</pre><br>&bull; <code>updated.title</code> &rarr; independent copy ✅<br>&bull; <code>updated.tags</code> &rarr; same array reference as <code>task.tags</code> ⚠️<br><br>Mutating <code>updated.tags.push(&quot;urgent&quot;)</code> would also change <code>task.tags</code>.<br><br>A <b>deep copy</b> (e.g. <code>structuredClone(task)</code>) creates fully independent nested copies.<br><br>For the <code>Task</code> interface in this course, spread is sufficient &mdash; no nested mutable structures.</div>	ch06 interfaces immutability concept
