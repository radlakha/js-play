<div class="q">What is the role of <code>ts-node</code> in a TypeScript project?</div><pre>npm install --save-dev typescript ts-node @types/node</pre><div class="options"><div class="option"><span class="opt-label">A)</span> It compiles TypeScript to JavaScript and saves output to <code>/dist</code></div><div class="option"><span class="opt-label">B)</span> It runs <code>.ts</code> files directly in the terminal without a separate compile step</div><div class="option"><span class="opt-label">C)</span> It provides type definitions for Node.js built-ins like <code>process</code></div><div class="option"><span class="opt-label">D)</span> It watches for file changes and reloads the app automatically</div></div>	<div class="answer">B &mdash; runs <code>.ts</code> files directly, no compile step needed</div>	ch00 tooling
<div class="q">Why is <code>@types/node</code> installed in a TypeScript + Node.js project?</div><pre>npm install --save-dev @types/node</pre><div class="options"><div class="option"><span class="opt-label">A)</span> It allows Node.js to understand TypeScript syntax</div><div class="option"><span class="opt-label">B)</span> It enables <code>ts-node</code> to compile files faster</div><div class="option"><span class="opt-label">C)</span> It provides TypeScript type definitions for Node.js built-ins like <code>process</code> and <code>console</code></div><div class="option"><span class="opt-label">D)</span> It is required to use <code>import</code> instead of <code>require</code></div></div>	<div class="answer">C &mdash; type definitions for Node.js built-ins</div>	ch00 tooling
<div class="q">What does <code>"strict": true</code> in <code>tsconfig.json</code> do?</div><pre>{<br>  "compilerOptions": {<br>    "strict": true<br>  }<br>}</pre><div class="options"><div class="option"><span class="opt-label">A)</span> Prevents the compiler from accepting any JavaScript files</div><div class="option"><span class="opt-label">B)</span> Forces all variables to be declared with <code>const</code></div><div class="option"><span class="opt-label">C)</span> Enables a set of strict type checks that catch more bugs at compile time</div><div class="option"><span class="opt-label">D)</span> Makes TypeScript reject code that uses <code>let</code></div></div>	<div class="answer">C &mdash; enables stricter type checks across the project</div>	ch00 config
<div class="q">What happens if <code>"types": ["node"]</code> is missing from <code>tsconfig.json</code> and you use <code>process.stdin</code>?</div><div class="options"><div class="option"><span class="opt-label">A)</span> Node.js throws a runtime error when the app starts</div><div class="option"><span class="opt-label">B)</span> TypeScript gives a compile error &mdash; it does not know what <code>process</code> is</div><div class="option"><span class="opt-label">C)</span> <code>process.stdin</code> works fine &mdash; TypeScript infers it automatically</div><div class="option"><span class="opt-label">D)</span> The app compiles but crashes silently when stdin is accessed</div></div>	<div class="answer">B &mdash; TypeScript won't recognise <code>process</code> without the Node types</div>	ch00 config nodejs
<div class="q">Which <code>tsconfig.json</code> setting controls where compiled JavaScript files are written?</div><pre>{<br>  "compilerOptions": {<br>    "rootDir": "./src",<br>    "outDir": "./dist"<br>  }<br>}</pre><div class="options"><div class="option"><span class="opt-label">A)</span> <code>rootDir</code></div><div class="option"><span class="opt-label">B)</span> <code>include</code></div><div class="option"><span class="opt-label">C)</span> <code>target</code></div><div class="option"><span class="opt-label">D)</span> <code>outDir</code></div></div>	<div class="answer">D &mdash; <code>outDir</code> sets the compiled output destination</div>	ch00 config
<div class="q">What does <code>npm run start</code> do given this config?</div><pre>"scripts": {<br>  "build": "tsc",<br>  "start": "ts-node src/todo.ts",<br>  "test": "jest"<br>}</pre><div class="options"><div class="option"><span class="opt-label">A)</span> Compiles TypeScript to JavaScript, then runs the output from <code>/dist</code></div><div class="option"><span class="opt-label">B)</span> Runs <code>src/todo.ts</code> directly using <code>ts-node</code> without compiling first</div><div class="option"><span class="opt-label">C)</span> Starts the Jest test watcher</div><div class="option"><span class="opt-label">D)</span> Runs the compiled file from <code>/dist</code></div></div>	<div class="answer">B &mdash; <code>ts-node</code> runs the file directly, no <code>tsc</code> step</div>	ch00 tooling
<span class="concept-label">Concept</span><div class="q">What is the job of <code>tsconfig.json</code>? Name three settings it controls.</div>	<div class="concept-answer"><b><code>tsconfig.json</code> &mdash; TypeScript Compiler Config</b><br><br>Controls how the TypeScript compiler behaves for the entire project.<br><br>Key settings:<br>&bull; <code>"strict"</code> &mdash; how aggressively TypeScript checks your code<br>&bull; <code>"outDir" / "rootDir"</code> &mdash; where source lives and where compiled JS goes<br>&bull; <code>"target"</code> &mdash; which JavaScript version to compile to<br>&bull; <code>"types": ["node"]</code> &mdash; enables Node.js globals like <code>process</code><br><br>Without it, the compiler uses defaults that are often too loose for real projects.</div>	ch00 config concept
