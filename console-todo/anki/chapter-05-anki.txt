<div class="q">What is the key difference between an acceptance test and a unit test?</div><div class="options"><div class="option"><span class="opt-label">A)</span> Acceptance tests are written by QA; unit tests are written by developers</div><div class="option"><span class="opt-label">B)</span> Acceptance tests cover entire user workflows and assert on observable behavior; unit tests focus on individual functions in isolation</div><div class="option"><span class="opt-label">C)</span> Acceptance tests run in production; unit tests only run locally</div><div class="option"><span class="opt-label">D)</span> Acceptance tests are faster to run than unit tests</div></div>	<div class="answer">B &mdash; behavior vs function isolation</div>	ch05 atdd testing

<span class="concept-label">Concept</span><div class="q">What does it mean to say acceptance tests are "living documentation"?</div>	<div class="concept-answer"><b>Acceptance Tests as Living Documentation</b><br><br>Traditional documentation (README, wiki) goes stale &mdash; code changes, docs don't.<br><br>Acceptance tests describe what the system <b>actually does</b> in runnable code. If behavior changes and tests aren't updated, they fail &mdash; alerting you immediately.<br><br>Example test title: <em>"User can toggle a task between complete and incomplete"</em><br><br>That title <i>is</i> the requirement. The test body shows exactly how it works. Unlike a wiki page, it cannot silently drift out of sync with reality.</div>	ch05 atdd testing concept
