<div class="q">What is the difference between <code>toEqual</code> and <code>toBe</code> in Jest?</div><pre>expect(result).toEqual(["Buy milk", "Walk dog"]);<br>expect(result).not.toBe(initial);</pre><div class="options"><div class="option"><span class="opt-label">A)</span> <code>toEqual</code> checks reference equality; <code>toBe</code> checks contents</div><div class="option"><span class="opt-label">B)</span> <code>toEqual</code> only works on strings; <code>toBe</code> works on all types</div><div class="option"><span class="opt-label">C)</span> <code>toEqual</code> checks deep content equality; <code>toBe</code> checks the two values are the exact same object in memory</div><div class="option"><span class="opt-label">D)</span> They are identical &mdash; <code>toBe</code> is just an older alias</div></div>	<div class="answer">C &mdash; use <code>toEqual</code> for contents, <code>toBe</code> for reference identity</div>	ch04 testing jest

<div class="q">What does <code>describe()</code> do in a Jest test file?</div><pre>describe('Todo App Core Functions', () => {<br>    test('addTask should ...', () => { ... });<br>    test('markTaskComplete should ...', () => { ... });<br>});</pre><div class="options"><div class="option"><span class="opt-label">A)</span> It runs the tests inside it in isolation from other test files</div><div class="option"><span class="opt-label">B)</span> It skips the enclosed tests unless explicitly run</div><div class="option"><span class="opt-label">C)</span> It groups related tests under a label, making output easier to read and organize</div><div class="option"><span class="opt-label">D)</span> It is required by Jest &mdash; tests cannot exist outside a <code>describe</code> block</div></div>	<div class="answer">C &mdash; grouping label only; tests inside still run normally</div>	ch04 testing jest
<span class="concept-label">Concept</span><div class="q">Why are pure functions easier to test than functions with side effects?</div>	<div class="concept-answer"><b>Pure Functions and Testability</b><br><br>A pure function takes inputs and returns an output &mdash; nothing else.<br><br>In a test you only need to pass in inputs and check the return value. No global state to set up, no <code>console.log</code> to mock.<br><br><pre>// Pure — trivially testable<br>function addTask(tasks, item) { return [...tasks, item]; }<br><br>// Impure — needs extra setup<br>function addTask(item) { globalTasks.push(item); }</pre><br>This is why the course separates pure functions from the interactive CLI layer.</div>	ch04 functional testing concept
