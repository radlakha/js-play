<div class="q">What is the Fragile Test Problem?</div><div class="options"><div class="option"><span class="opt-label">A)</span> Tests that fail randomly due to timing issues in async code</div><div class="option"><span class="opt-label">B)</span> Tests that break when internal implementation details change, even though user-facing behavior has not changed</div><div class="option"><span class="opt-label">C)</span> Tests that pass even when the code is wrong</div><div class="option"><span class="opt-label">D)</span> Tests that are too slow to run in a CI pipeline</div></div>	<div class="answer">B &mdash; breaks on internal changes unrelated to behavior</div>	ch05a atdd testing
<div class="q">After refactoring <code>tasks</code> from <code>string[]</code> to <code>Task[]</code>, what happens to this test?</div><pre>let tasks: string[] = [];<br>tasks = addTask(tasks, "Buy milk");<br>tasks = toggleTaskCompletion(tasks, 1);<br><br>expect(tasks).toEqual([<br>    "Buy milk",<br>    "[✓] Walk the dog",<br>]);</pre><div class="options"><div class="option"><span class="opt-label">A)</span> Nothing &mdash; acceptance tests are immune to internal data shape changes</div><div class="option"><span class="opt-label">B)</span> The test still passes &mdash; TypeScript automatically converts between types</div><div class="option"><span class="opt-label">C)</span> The test breaks in two places: the <code>string[]</code> annotation and the raw array assertion &mdash; both depend on internal shape</div><div class="option"><span class="opt-label">D)</span> Only the <code>expect</code> line breaks &mdash; the Arrange section is unaffected</div></div>	<div class="answer">C &mdash; both the type annotation and the assertion are coupled to internals</div>	ch05a atdd testing
<div class="q">What is the purpose of the Application API layer?</div><pre>export function createTask(title: string): number { ... }<br>export function toggleTask(id: number): void { ... }<br>export function isTaskComplete(id: number): boolean { ... }</pre><div class="options"><div class="option"><span class="opt-label">A)</span> To replace pure functions with object-oriented classes</div><div class="option"><span class="opt-label">B)</span> To add UUID-based IDs to every task</div><div class="option"><span class="opt-label">C)</span> To create a stable boundary that hides internal data shape from callers, so acceptance tests survive refactoring</div><div class="option"><span class="opt-label">D)</span> To prevent the CLI from accessing the task list directly</div></div>	<div class="answer">C &mdash; stable boundary; internals can change without breaking callers</div>	ch05a atdd architecture
<div class="q">Why should acceptance tests always use the returned <code>id</code> from <code>createTask</code>?</div><pre>// GOOD — treats id as opaque<br>const id = createTask("Walk the dog");<br>toggleTask(id);<br><br>// BAD — assumes id === 1<br>createTask("Walk the dog");<br>toggleTask(1);</pre><div class="options"><div class="option"><span class="opt-label">A)</span> The returned id is always a UUID, not a number</div><div class="option"><span class="opt-label">B)</span> TypeScript will not compile the BAD version</div><div class="option"><span class="opt-label">C)</span> Hardcoding <code>1</code> couples the test to the current implementation &mdash; if ID generation changes, every hardcoded assumption breaks</div><div class="option"><span class="opt-label">D)</span> <code>toggleTask(1)</code> is invalid syntax</div></div>	<div class="answer">C &mdash; treat ids as opaque; never assume id equals a specific value</div>	ch05a atdd testing
<div class="q">Why does this course choose a module with module-level state over a class?</div><pre>// Chosen: functional module<br>let appTasks: string[] = [];<br>export function createTask(title: string): number { ... }<br><br>// Not chosen: OOP class<br>class TodoService {<br>    private tasks = [];<br>    addTask(title: string) { ... }<br>}</pre><div class="options"><div class="option"><span class="opt-label">A)</span> Classes are not supported in TypeScript</div><div class="option"><span class="opt-label">B)</span> The class approach requires a database connection</div><div class="option"><span class="opt-label">C)</span> This course focuses on functional programming &mdash; pure functions, immutability, no hidden state via <code>this</code></div><div class="option"><span class="opt-label">D)</span> Module-level state is faster at runtime than class instances</div></div>	<div class="answer">C &mdash; functional style avoids <code>this</code> and encapsulated mutable state</div>	ch05a architecture functional
<div class="q">What does YAGNI mean and how does it apply here?</div><div class="options"><div class="option"><span class="opt-label">A)</span> You Always Get Needed Improvements &mdash; add UUIDs now to prevent future work</div><div class="option"><span class="opt-label">B)</span> You Aren't Gonna Need It &mdash; use the simplest solution that satisfies current requirements; add UUID complexity only when a requirement explicitly demands it</div><div class="option"><span class="opt-label">C)</span> Your API Generates New Identifiers &mdash; a naming convention for ID fields</div><div class="option"><span class="opt-label">D)</span> YAGNI only applies to features, not to data types</div></div>	<div class="answer">B &mdash; simplest solution now; complexity only when a real requirement demands it</div>	ch05a architecture
<span class="concept-label">Concept</span><div class="q">After Chapter 05a the codebase has two distinct layers. What is the job of each?</div>	<div class="concept-answer"><b>Two Layers: Pure Functions vs Application API</b><br><br><b>Pure functions layer</b> (from earlier chapters):<br>&bull; <code>addTask</code>, <code>toggleTaskCompletion</code>, etc.<br>&bull; No state &mdash; takes inputs, returns new values<br>&bull; Tested by unit tests<br><br><b>Application API layer</b> (new in 05a):<br>&bull; <code>createTask</code>, <code>toggleTask</code>, <code>isTaskComplete</code>, <code>resetApp</code><br>&bull; Owns module-level state (<code>appTasks</code>)<br>&bull; Delegates to pure functions<br>&bull; Tested by acceptance tests<br><br>Key insight: acceptance tests attach to the <b>Application API</b>, not pure functions. Internal shape changes (<code>string[]</code> &rarr; <code>Task[]</code>) stay invisible to them.</div>	ch05a architecture testing concept
